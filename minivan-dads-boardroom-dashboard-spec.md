# Spec: Boardroom Protocol + CEO Dashboard — Minivan Dads Inc.

**Handoff document for Claude Code. Companion to the project brief and action-layer spec.**
**Reference UX: `ceo-dashboard-demo.html` (dashboard + boardroom) and `board-meeting-demo.html` (meeting). Make the real thing feel like these.**

---

## Part A — Boardroom (multi-agent deliberation)

### A1. Command

`brain boardroom "<topic>"` — convene a debate among relevant departments on any topic. Also triggerable from the dashboard, and the brain may PROPOSE a boardroom topic in the weekly agenda when department reports conflict.

### A2. How participants work (this is the important part)

Each participant is a **separate model call with its own context** — not the brain doing voices:

- Participant context = company charter + that department's charter + its current directive + its last 2 reports + the topic + the debate transcript so far.
- The brain (moderator) selects which departments are relevant to the topic (config override allowed: `--all` or `--depts creative,finance`).
- Dormant departments may be summoned in "advisory" mode (no directive/report context yet — reasoning from charter only, flagged as such).

### A3. The honesty norm (charter amendment — add to `hq/charter/company.md`)

> **Boardroom conduct:** Every participant argues for what is best for the COMPANY, seen from its department's vantage point. Departments do not defend turf, budgets, or past work. Disagreement on the record is required when it exists; consensus-seeking and deference to the CEO's presumed preference are charter violations. A dissent recorded and overruled is a job well done.

Enforce structurally, not just by prompt: participant prompts must NOT include the CEO's stated lean on the topic during the opening-positions round. The CEO's view enters the transcript only after initial positions are filed.

### A4. Debate structure (rounds)

1. **Positions** — each participant files an opening position (≤150 words) with its single strongest reason. Written in parallel (no participant sees the others' positions in this round — prevents anchoring).
2. **Rebuttal** — each participant sees all positions and may rebut or update (updating with stated reason is encouraged; silent flips are flagged by the moderator).
3. **CEO floor** — interactive. The CEO may question any participant by name, challenge, or add constraints. Addressed participant responds with full context; others may interject only if the brain rules it relevant.
4. **Synthesis** — the brain produces a recommendation that names the strongest objection and how it's handled (as a gate, a condition, or an accepted risk). The brain may recommend AGAINST the majority.
5. **Ruling** — CEO decides. Log entry must include: decision, rationale, each department's final position, and dissents by name. Directives affected are updated immediately.

Output artifacts: `hq/meetings/{week}-boardroom-{slug}.md` (full transcript + ruling), decision-log entries, directive updates.

### A5. Guardrails

- Boardroom produces decisions and directives, never direct actions — anything actionable flows through the normal executor path afterward.
- Max 2 rebuttal rounds; the brain must call the question (debates that don't end aren't deliberation, they're token burn).
- Cost note: a 5-participant debate is ~12–20 model calls. Fine weekly; the brain should decline to convene the full boardroom for questions `brain ask` can answer alone.

## Part B — CEO Dashboard

### B1. Architecture

`brain dashboard` serves a **local web app** (FastAPI + static frontend, or equivalent) that live-reads HQ. No cloud, no auth for v1 — it binds to localhost. The dashboard is a **view over HQ files plus a chat surface**; it holds no state of its own. Anything visible in the dashboard must correspond to a file in HQ (single source of truth preserved).

### B2. Views (match the demo's four tabs)

1. **Dashboard** — stat row (open escalations, reports filed this week, decision count, stale directives), escalation inbox, recent decisions, "this week from the brain" (generated by `brain ingest`, cached).
2. **Departments** — card grid (name, tier pill, status, last report date) → drill-down per department: standing directive, latest report, actions-taken log (from `hq/actions/log.jsonl`), directive version history (git log of the file).
3. **Boardroom** — open a topic (free text or brain-suggested), watch positions stream in, participate via text box (replaces the demo's canned chips), rule, see the logged decision.
4. **Commands** — live reference generated from the CLI's own help text, so it never drifts from reality.

### B3. Interaction surfaces

- **Hard requirement — every decision point offers both:** wherever suggested rulings appear (chips/buttons in the dashboard, numbered options in the CLI), a free-text input is ALWAYS also available. Suggestions are accelerators, never the only path. In the CLI: typing anything other than a listed option is treated as discussion/a question addressed to the brain (or `@department` to address an agent in the boardroom). In the dashboard: chips render above a persistent text box.
- **Voice:** v1 comes free via OS dictation into the text box (mobile keyboard mic, macOS/Windows dictation) — build nothing. A later upgrade may add in-app speech-to-text (e.g., Whisper) behind the same text input; do not build it before the core loop works.
- Boardroom and `brain ask` are available as chat inputs in the dashboard (streaming responses).
- `brain meeting` may run in the dashboard as a guided flow (agenda item cards with ruling buttons + free-text discuss), equivalent to the CLI version — same artifacts written either way.
- Escalation inbox items get inline actions: "handle now" (opens brain ask with context preloaded), "defer to Friday", "dismiss with note".

### B4. Build order

1. Read-only dashboard (tabs 1, 2, 4) over HQ files — no model calls, ships fast.
2. `brain ask` chat in the dashboard.
3. Boardroom protocol in the CLI (transcript to file), then surfaced in the dashboard with streaming.
4. Meeting-in-dashboard last (CLI meeting already covers the need).

**Acceptance test:** CEO runs `brain dashboard`, opens the Boardroom tab, enters a real topic; four departments file genuinely different positions (verify: run it on a topic where Finance and Creative should conflict — if everyone agrees, the honesty norm is failing); CEO questions one participant by name and gets a context-aware answer; CEO rules; transcript, decision-log entry with named dissents, and directive updates all appear in HQ and in the Departments tab without a restart.
